# Autoshop Takumi — Bilingual Vehicle Sales Website

A bilingual (Japanese/English) vehicle sales website for Autoshop Takumi, built as a reusable template for other dealerships. Visitors can browse and filter inventory in either language; non-technical staff manage all content from a generated admin dashboard — no code changes required.

---

## Stack

| Layer | Technology |
|---|---|
| Framework | Next.js 15 (App Router, React Server Components) |
| CMS / Admin | Payload CMS 3.x (embedded in the Next.js app) |
| Database | SQLite (local dev) → PostgreSQL/Neon (production) |
| Image storage | Local (dev) → Cloudflare R2 (production) |
| UI components | shadcn/ui + Tailwind CSS v4 |
| i18n (static UI) | next-intl (`/ja/...`, `/en/...` URL routing) |
| i18n (content) | Payload field-level localization (JA + EN per field) |
| Email | Resend (inquiry notifications to staff) |
| Hosting | Vercel |

## What's Built

### Public site

| Page | Route | Description |
|---|---|---|
| Landing | `/[locale]` | Hero, about blurb, featured vehicles, why-us section, contact summary |
| Vehicle listing | `/[locale]/vehicles` | Filterable + sortable grid; filters: make, model, year, price, body type, transmission |
| Vehicle detail | `/[locale]/vehicles/[slug]` | Photo gallery with lightbox, highlights list, flexible spec table, inquiry form, related vehicles |

The language switcher in the header swaps both the UI chrome (nav, buttons, labels) and the vehicle content (titles, descriptions, spec table rows) simultaneously. URLs are locale-prefixed so both versions are independently linkable and indexable by search engines.

### Admin dashboard (`/admin`)

Entirely auto-generated by Payload CMS from the schema in `payload.config.ts` — no bespoke admin screens were written:

- **Vehicles** — full CRUD, draft/published workflow, bilingual field tabs (JA/EN), image upload with drag-and-drop, gallery reorder, clone-listing helper
- **Makes / Models** — editable taxonomy; adding a new make/model here immediately makes it available as a listing-page filter with no redeploy
- **Media** — image library with automatic resizing to 4 size variants (thumbnail/card/detail/full) on upload
- **Inquiries** — inbox of contact-form submissions with New/Contacted/Closed status tracking
- **Site Settings** — shop name, logo, contact info, social links, notification email addresses
- **Homepage** — hero copy/image, about blurb, featured vehicle picks, why-us points — all editable without a deploy

```
Inquiries ──→ Vehicles

SiteSettings  (global singleton)
Homepage      (global singleton)
Users         (auth collection)
```

### Vehicles collection (key fields)

| Field | Type | Notes |
|---|---|---|
| `title` | text (localized) | e.g. "1999 Toyota Supra RZ" / "1999 トヨタ スープラ RZ" |
| `slug` | text, unique | URL slug, auto-generated from EN title |
| `status` | select | `draft` / `available` / `reserved` / `sold` |
| `make` / `model` | relationship | Editable taxonomy — not a hardcoded enum |
| `year`, `mileageKm` | number | |
| `price` / `priceOnRequest` | number / checkbox | When `priceOnRequest` is true, shows "Contact for price" |
| `transmission` / `bodyType` | select | MT/AT/CVT; sedan/coupe/SUV/wagon/kei |
| `shakenExpiry` | date | 車検 expiry — hidden on frontend if blank |
| `summary` | textarea (localized) | Short overview at top of detail page |
| `highlights` | array (localized) | Bullet "POINT" list, matches reference site pattern |
| `description` | richText (localized) | Long narrative |
| `specs` | array of label/value (localized) | Flexible spec table — staff add/remove rows per vehicle |
| `heroImage` / `gallery` | upload → Media | Required before a vehicle can be published |
| `featured` | checkbox | Surfaces on homepage featured section |
| `seoTitle` / `seoDescription` | text (localized) | Per-vehicle SEO overrides |

A `beforeChange` hook blocks setting `status: available` if `heroImage` is missing, so incomplete listings can't go live by accident.

## Design System

### Brand tokens

| Token | Approx. value | Use |
|---|---|---|
| `--brand-orange` | `#F26A21` (re-sample from real logo) | Primary actions, active states, accent bars |
| `--brand-black` | `#111111` | Headlines, nav background |
| `--brand-white` | `#FFFFFF` | Page background, reversed text |
| `--status-available` | `#1A7F37` (green) | "Available" badge |
| `--status-reserved` | `#B58105` (amber) | "Reserved" badge |
| `--status-sold` | `#6B6B6B` (gray) | "Sold" badge — muted, not alarming, since sold history is a credibility signal |

All tokens are CSS variables in `src/app/globals.css`, not hardcoded utility classes — swapping them once is enough to rebrand for a second shop.

### Typography

- **Headings:** condensed bold sans (Archivo Black / Oswald) for Latin; falls back to body font for Japanese (condensed Latin faces rarely have kanji coverage)
- **Body / UI:** Noto Sans JP + Inter, loaded via `next/font` (self-hosted, no render-blocking third-party request)

## Local Development

**Prerequisites:** Node 20+, no other accounts needed for local dev.

```bash
# 1. Clone and install
git clone https://github.com/chamirusenarath96/autoshop-takumi.git
cd autoshop-takumi
npm install

# 2. Set up environment
cp .env.example .env
# Edit .env: set PAYLOAD_SECRET to any random string

# 3. Start dev server (SQLite database is created automatically)
npm run dev
# → http://localhost:3000       public site
# → http://localhost:3000/admin  admin dashboard

# 4. (Optional) Seed sample data
npm run seed
```

On first visit to `/admin`, Payload will prompt you to create the first admin user.

## Environment Variables

```bash
# Required
NEXT_PUBLIC_SITE_URL=http://localhost:3000
PAYLOAD_SECRET=                    # Any random secret string

# Production: PostgreSQL (Neon recommended)
DATABASE_URI=postgres://user:pass@host/db?sslmode=require

# Production: Cloudflare R2 image storage
R2_BUCKET=
R2_ENDPOINT=                       # https://<account>.r2.cloudflarestorage.com
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_PUBLIC_URL=                     # Public bucket URL

# Production: Resend email (inquiry notifications)
RESEND_API_KEY=
NOTIFY_EMAIL_FROM=inquiries@autoshoptakumi.example
NOTIFY_EMAIL_TO=staff@autoshoptakumi.example
```

Local dev works with only `PAYLOAD_SECRET` set. All other variables are production-only.

## Production Deployment (Vercel)

1. Create accounts: [Neon](https://neon.tech) (Postgres), [Cloudflare R2](https://developers.cloudflare.com/r2/) (storage), [Resend](https://resend.com) (email)
2. Connect this GitHub repo to a new Vercel project
3. Add all environment variables in Vercel project settings
4. Switch the `db` adapter in `payload.config.ts` from `sqliteAdapter` to `postgresAdapter` and add the `@payloadcms/storage-s3` plugin for R2
5. Push to `main` — Vercel auto-deploys

## Reusing as a Template (Shop #2)

Because all shop identity lives in Payload data rather than component code, a second shop is stood up by:

1. Fork this repo
2. Point it at a new database and R2 bucket (new env vars)
3. Update the brand tokens in `src/app/globals.css`
4. Seed new `SiteSettings` (shop name, logo, contact info) and `Makes`/`Models` taxonomy
5. Deploy

No page templates, filter logic, or admin screens need to be rewritten.
